---
title: "TLS certificates"
description: "Understand fake-gcs-server's default self-signed certificate, how to provide a custom certificate, and when to use HTTP instead of HTTPS."
---

fake-gcs-server defaults to HTTPS using a self-signed TLS certificate generated by Go's `net/http/httptest` package. This certificate is not trusted by system certificate stores, so clients need explicit configuration to connect.

## Default self-signed certificate

When you start the server without providing certificate files, it generates a self-signed certificate in memory at startup. The certificate is unique to each server process and is not persisted to disk. This is the simplest way to get started, but every HTTP client that connects must be told to skip certificate verification.

<Warning>
  Disabling certificate verification (`InsecureSkipVerify`, `--insecure`) is safe in local development and testing, but must never be used in production code paths.
</Warning>

### curl

Use the `--insecure` flag to skip verification:

```shell
curl --insecure https://0.0.0.0:4443/storage/v1/b
```

### Go

Set `InsecureSkipVerify` on the TLS config used by your HTTP client:

```go
import (
    "crypto/tls"
    "net/http"

    "cloud.google.com/go/storage"
    "google.golang.org/api/option"
)

httpClient := &http.Client{
    Transport: &http.Transport{
        TLSClientConfig: &tls.Config{InsecureSkipVerify: true},
    },
}

client, err := storage.NewClient(ctx, option.WithHTTPClient(httpClient))
```

Alternatively, if you are using the `fakestorage` Go package directly in your tests, call `Server.Client()` or `Server.HTTPClient()` — these return a pre-configured client that already trusts the server's certificate:

```go
server := fakestorage.NewServer(nil)
defer server.Stop()

// Client() returns a *storage.Client pre-wired to the fake server.
gcsClient := server.Client()
```

## Providing a custom certificate

If you need clients to trust the server's certificate without disabling verification — for example, when running the emulator in a CI environment that has a shared CA — you can supply your own certificate and private key.

Use the `-cert-location` and `-private-key-location` flags:

```shell
fake-gcs-server \
  -cert-location /path/to/server.crt \
  -private-key-location /path/to/server.key
```

Or with Docker:

```shell
docker run -d \
  --name fake-gcs-server \
  -p 4443:4443 \
  -v ${PWD}/certs:/certs:ro \
  fsouza/fake-gcs-server \
  -cert-location /certs/server.crt \
  -private-key-location /certs/server.key
```

The certificate and key must be in PEM format and must be a valid pair. The server calls `tls.LoadX509KeyPair` on startup and will refuse to start if the files cannot be loaded.

You can also use environment variables:

```shell
export FAKE_GCS_CERT_LOCATION=/path/to/server.crt
export FAKE_GCS_PRIVATE_KEY_LOCATION=/path/to/server.key
fake-gcs-server
```

### Generating a self-signed certificate for a custom CA

If you manage a local CA for your development environment, generate a certificate with:

```shell
# Generate a private key
openssl genrsa -out server.key 2048

# Generate a self-signed certificate valid for 365 days
openssl req -new -x509 \
  -key server.key \
  -out server.crt \
  -days 365 \
  -subj "/CN=fake-gcs-server" \
  -addext "subjectAltName=IP:127.0.0.1,DNS:localhost"
```

Add `server.crt` to your system or container's trusted CA store and clients will connect without disabling verification.

## Running with HTTP only

For local development and simple test setups, you can avoid TLS entirely by running the server in HTTP mode. This removes the need for `--insecure` flags and `InsecureSkipVerify` configuration.

Pass `-scheme http` to switch to plain HTTP:

<Tabs>
  <Tab title="Docker">
    ```shell
    docker run -d \
      --name fake-gcs-server \
      -p 4443:4443 \
      -v ${PWD}/examples/data:/data \
      fsouza/fake-gcs-server \
      -scheme http
    ```
  </Tab>
  <Tab title="CLI / binary">
    ```shell
    fake-gcs-server -scheme http
    ```
  </Tab>
  <Tab title="Environment variable">
    ```shell
    docker run -d \
      --name fake-gcs-server \
      -p 4443:4443 \
      -e FAKE_GCS_SCHEME=http \
      fsouza/fake-gcs-server
    ```
  </Tab>
</Tabs>

With HTTP, curl requests work without any extra flags:

```shell
curl http://0.0.0.0:4443/storage/v1/b
```

```json
{"kind":"storage#buckets","items":[{"kind":"storage#bucket","id":"sample-bucket","name":"sample-bucket"}],"prefixes":null}
```

## Running both HTTPS and HTTP simultaneously

Pass `-scheme both` to start HTTPS and HTTP listeners at the same time. HTTPS binds to `-port` (default `4443`) and HTTP binds to `-port-http` (default `8000`):

```shell
docker run -d \
  --name fake-gcs-server \
  -p 4443:4443 \
  -p 8000:8000 \
  -v ${PWD}/examples/data:/data \
  fsouza/fake-gcs-server \
  -scheme both
```

This is useful when you have a mix of clients: some configured for HTTPS and others for HTTP.
